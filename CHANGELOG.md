## [Unreleased]

### Added - #8: E2E artifact-failure injection and hook output-contract enforcement (the exit-1 bug class)
- **The failure paths of the gate machine had never been driven end to end.** The happy-path suites proved a valid worker return advances the gate; nothing proved what happens when a return is malformed, empty, prose-only, or misattributed, when an artifact half-derives, or when the hook itself dies mid-advance. `test/e2e/failure-injection.e2e.test.mjs` (new) drives the real `hooks/gate-advance.mjs`, `hooks/contract-validator.mjs`, and `hooks/subagent-budget-guard.mjs` as subprocesses against a temp-isolated monoroot workspace, one injected fault per row, and asserts three things every time: the gate is exactly where it should be, no partial artifact was left on disk, and the failure reached a host-honored surfacing channel.
- **Every invalid worker return is proven to advance nothing, loudly.** A table drives malformed router/grill/critique/planner/diagnose returns plus the empty, prose-only, and unattributed dispatches; each writes no canonical artifact, leaves the gate put, and blocks with a model-visible explanation (exit 2) that names who failed and forbids doing the work inline. The discovery fan-in is deliberately excluded — it tolerates a single bad worker by design (skips it, still merges), and that tolerance is covered by the merge's own unit tests.
- **The exit-code contract is asserted against the real host semantics.** The contract validator blocks with exit 2 — the only non-zero code VS Code treats as blocking; exit 1 is a non-blocking warning whose stdout the host never reads. The suite pins exit 2 for a malformed artifact, exit 0 for a valid one and for an unrouted edit, and `docs/hooks.md` is corrected in three places where it wrongly claimed exit 1 blocked.
- **A spec that never landed, and a plan that yields no scope, are both proven safe.** An empty or wrong-path spec does not open the human review gate; a plan whose only task lists no files writes `plan.json` but refuses an empty `scope.md` (a partial projection, reported on stderr), and the implementation dispatch is then denied for want of the scope contract.
- **A crash mid-advance leaves the gate unmoved, surfaces model-visible catch-up guidance, and the next call heals it.** A deterministic, test-only fault seam (`lib/testing/fault-injection.mjs`, new) faults `hooks/gate-advance.mjs` after the artifact is written atomically but before the gate walks — proving the gate never half-moves and the next unarmed invocation catches up. The caught crash exits `0` and emits its recovery guidance on the documented `additionalContext` channel the model reads (the raw error stays on stderr for the human Output panel, with no stack), so a recoverable stall no longer reads as a clean, successful no-op turn. A timeout variant (HARNESS-EMULATED: a real host kills a hung hook; the harness's short spawn timeout stands in) can emit nothing after SIGTERM, so its model visibility comes from the NEXT invocation's catch-up anchor — the suite asserts exactly that and never claims same-turn visibility for a kill. A broken trace write (a directory where the audit line should go) is a scoped best-effort failure that does not roll back the gate it already recorded, and the gate still emits its normal advance anchor while a trace-specific warning is loud on stderr. Both crash and timeout also assert the artifact is written whole (parseable, no partial `.tmp` sibling), not merely present.
- **The seam is impossible to arm from production.** It is inert unless its env var names a known site and mode, the asking call site is itself a known site, reads only the injected environment, throws only its own typed error, and does one Set lookup when unarmed. `test/lib/testing/no-production-fault.test.mjs` (new) enforces that no shipped file under `lib/`, `hooks/`, or `scripts/` even names the variable outside the seam module, and `test/lib/testing/fault-injection.test.mjs` (new) pins the seam's grammar and modes.
- **Pre-existing integration-baseline repairs, so `npm run verify` is genuinely green.** Three fixes that predate this suite but block a green PR on the stacked branch: (1) a `check-docs-drift` false-positive on a backticked type name (the TurnResult identifier) in the #131 CHANGELOG entry, resolved by de-backticking it to plain prose; (2) a cross-test workspace leak where `test/lib/workflow/lanes/bug.test.mjs`'s `runBugHandoff` regression test wrote `bug-42` state into the repo tree via `dispatchFixer`'s cwd-relative default paths — fixed by isolating the write to a temp dir with an isolation regression assertion; and (3) `scripts/check-artifact-allowlist.mjs` ran in CI but not in `npm run verify`, so tree contamination failed only in CI — now wired into `verify` so it fails locally too, matching CLAUDE.md's "green locally means a green PR".

### Added - #6: an E2E deny matrix proving every reachable PreToolUse gate-guard denial is actionable and recoverable
- **The deny ladder was only ever asserted from the inside.** Unit tests called the pure decision function directly, so they proved the evaluator's verdict but never that the real `scripts/gate-guard.mjs` subprocess — the one the host actually runs on `PreToolUse` — surfaces that verdict through the channel the host reads. `test/e2e/gate-guard-deny-matrix.e2e.test.mjs` (new) drives the real hook as a subprocess against a temp-isolated monoroot workspace, one perturbed dimension per row, and reads the decision the host would act on rather than a mirror of the ladder's own reason strings.
- **Every reachable rule×lane×gate deny is covered once, and every deny is recoverable.** A single allow-ready baseline is perturbed along exactly one axis per row to isolate each rule: missing config, unreadable state, budget-critical, the not-started gate (with its lane-aware guidance for feature/bug/chore), the terminal-as-editor and unrecognized-tool fail-closed branch, the session-artifact protection over `.devmate/state/**` and `.devmate/session/**`, the out-of-scope and no-scope refusals, and the TDD pre-condition. Each deny row carries a corrected follow-up call that the same workspace then ALLOWS — so a denial that strands the caller with no working escape fails the suite.
- **First-match ordering is pinned as precedence pairs.** Where two rules would both fire, the suite asserts the winning rule's guidance is present and the loser's is absent, proving the ladder resolves in the fixed order rather than by accident.
- **The deny contract is asserted, not just the verdict.** Every denial must name the offending path, direct the caller to the workflow's own recovery (`devmate init`, an approved spec, a scope contract, a failing test first) rather than leaking devmate's library source, and stay bounded (a scope with many globs elides to a `+N more` summary under the output cap). Non-edit and unknown/MCP tools that name no protected path stay allowed (fail-open), the golden-path allow carries no reason, and the guard answers within the timeout `hooks/hooks.json` registers.
- **The implementation-dispatch gate is covered as its own deny class.** A `runSubagent` call targeting the `fullstack` implementation agent is denied by `evaluateImplementationDispatch` — which runs *before* the rule ladder — until the lane's preconditions hold: a task in flight at `impl-started`, an approved spec (feature) or valid `diagnosis.json` (bug), and a `scope.md`. Each row withholds one precondition and asserts the deny, then a corrected twin dispatch is ALLOWED. The `agentName` the guard reads is a field of the *tool's* input schema (repo contract), not a host-verified payload field, and the rows say so; the structural second layer is the SubagentStart budget guard.
- **Every deny with a task context is now audited, and recoveries are not.** The guard appends one bounded, content-free `action` trace event per denial (via the same `auditAction`/`appendTraceEvent` path peer hooks use): the event carries the source layer and tool in its `actionType` and the offending path, and a digest — never the free-text reason and never file content. A deny with no readable task state has no task id to key a trace on, so it writes nothing while the host deny still stands; the suite pins both, and that a corrected (allowed) follow-up adds no deny event.

### Added - #5: gate-evidence consistency detection and an opt-in recovery path for hand-tampered task state
- **A hand-edited `workflowGate` was trusted on sight.** Nothing checked that the persisted gate was actually backed by the artifacts and audit events it legally requires, so editing `task.json` to `spec-approved` with no spec and no audited transition, or resetting the gate behind a trace that already advanced, silently authorized (or re-ran) work. `lib/gate-consistency.mjs` (new) makes advancement a pure function of what is on disk: `analyzeGateConsistency` classifies every divergence — forward (gate ahead of evidence), backward (gate behind the trace), forged (a human-audit gate reached with no audited transition), and malformed-trace (a corrupt `trace.jsonl` above a 5% line threshold) — and names the last evidence-backed gate to roll back to. `readTraceConsistency` reads the trace tolerantly (a few bad lines never crash the reader) and `checkGateConsistency` wraps the pure core with the on-disk reads.
- **Detection surfaces everywhere the user already looks.** The real `SessionStart` and `UserPromptSubmit` anchor paths now run the check and, when a divergence is found, add one `state: desynced` line to the model-visible anchor that names the divergence, the rollback target gate, and states dispatch stays denied until reconciled. `scripts/devmate-doctor.mjs` runs the same check as a stage: it exits 1 on an unreconciled divergence, prints the recovery command on stderr, and — only behind the opt-in `--fix` flag — reconciles the gate to the last evidence-backed gate under the state lock, stamps an audited `gate_transition` recording the reconcile, and re-checks so the healed result is what it reports. Detection is always non-destructive; the rewrite is opt-in.
- **Hooks fail safe, never into a crash.** A corrupt or deleted `task.json`, or a stale resume pointer naming a task that does not exist, never blocks a hook (no exit 2) and never throws — the doctor treats an unreadable state file as a no-op it cannot reconcile rather than an error. `test/e2e/state-tampering.e2e.test.mjs` (new) drives the real hooks and the real doctor as subprocesses against a temp workspace, tampering by hand exactly as a user would, and `test/lib/gate-consistency.test.mjs` (new) exercises the pure core across every divergence class plus a cross-check that the evidence chain never drifts from the canonical transition table.

### Added - #3: an E2E suite for the human-gate protocol under NON-happy input — wrong-gate approvals, rejection/revision loops, deferrals, read-only questions, post-approval regret, and trailing prose
- **The happy path was the only path anyone had tested.** A human typing the exact phrase at exactly the right gate was covered; everything a real human actually does at a decision gate was not. `test/e2e/human-gate-protocol.e2e.test.mjs` (new) drives `hooks/approval-listener.mjs` (and, for the rollback, `hooks/spec-integrity-guard.mjs`) as REAL subprocesses through the session harness, with the payload the VS Code host delivers on stdin and the workspace's own `.devmate/` as cwd. The oracle is the state that lands on disk (`task.json`, the JSONL trace, `turn-intent.json`) and the single `additionalContext` envelope the hook writes to stdout — never a mirror of the hook's own transition table.
- **The wrong-gate matrix is empirical, not copied.** For every `(lane, gate)` a human might sit at, all three canonical approval phrases (`approve spec` / `approve pr` / `approve plan`) are submitted; each must do exactly one of: advance the gate it legally can, be a friendly idempotent no-op, or REFUSE with model-visible guidance that names the legal next gates — and never wedge the workflow or silently pretend a gate moved. The legal-advance cells were confirmed against the real hook, so a test re-encoding the table would pass against a hook that stopped calling it.
- **Rejection, deferral, and read-only turns are pinned.** Three `revise spec:` cycles keep the gate at `spec-draft` and record verbatim feedback (including punctuation and colons) with no `spec_invalidated`; free-form and ambiguous input (`actually it should also handle SSO`, `hmm ok I guess?`) DEFERS at the deterministic Stage 1 (`turn-intent.json` carries `deferred:true`, `intent:null`) and never advances; questions/status at every human gate leave `task.json` byte-identical and write no trace.
- **Post-approval regret rolls back, then re-approval works.** Editing an approved `spec.md` on disk fires the real `spec-integrity-guard` on `PostToolUse`, which rolls `spec-approved → spec-draft` with a `spec_invalidated` + `gate_transition` audit pair; a `revise spec:` after the rollback records feedback without moving the gate; re-approving the rolled-back spec resumes the workflow into `impl-started`.
- **Trailing prose does NOT approve (a pinned decision).** `approve spec — but rename the module later` is not the exact phrase, so the deterministic layer emits a near-miss hint (`Did you mean: approve spec?`) and defers the turn to the LLM stage rather than opening implementation, because a gate is devmate's one human checkpoint and a trailing afterthought is exactly the ambiguity that must reach a human/LLM.
- **Both layers see one corpus, and they agree.** `evals/gate-robustness/fixtures/` gains the same revision phrasings and the status question (the `hgp-*` cases) so the eval's simulated Stage-2 interpreter is exercised on identical text; the interpreter's informational-question opener now recognises `what is`/`what are`. The trailing-prose approval is now part of the shared corpus: dropping bare `later` from the interpreter's non-blocking lexicon makes Stage 2 classify `approve spec — but rename the module later` as a change request (a deferred change is still a pending change), so it never reads as an approve-now nit. Both layers refuse to auto-advance the sole human gate on an inexact approval — no divergence is left latent.

### Added - #131: a scripted-turn driver and a stuck-state assertion helper for the E2E session harness
- **The harness could replay events but not a *turn*.** `test/e2e/session-harness.mjs` already ran the real hooks as real subprocesses against an unseeded monoroot workspace, but a scenario author had to hand-assemble every payload and eyeball the result. `runSession` adds the missing layer on top of the existing primitives (never replacing them): it bootstraps its own task state with a `SessionStart`, then replays each turn — a `UserPromptSubmit`, then its tool calls and subagent returns in order — and returns one typed TurnResult per turn (the hooks it ran, the state after, the gate).
- **The invariant this whole family of suites protects now has an assertion.** `isUserStuck` answers the question every scenario turns on: after this turn, could any real runtime caller have advanced the gate, or is the user wedged? - **Subagent returns get authored from the captured shape, never a guess.** `subagentReturnPayload` builds a `runSubagent` `PostToolUse` payload matching `test/fixtures/hook-payloads/captured/posttooluse.run-subagent.json` key-for-key: `tool_input` is the elided literal, and `tool_response` is the agent's chat text — prose followed by the embedded JSON contract, which is exactly where the gate-advance hook reads the agent's identity and result from.
- **The harness tests itself.** `test/e2e/session-harness.test.mjs` pins the payload against the captured fixture (deep-equal on keys), round-trips it through the real `extractAgentResult`, exercises `isUserStuck`'s stuck/not-stuck/genericity cases against hand-built states, and smoke-tests `runSession` over a two-turn script. All four existing `*.e2e.test.mjs` suites pass unchanged.

### Fixed - #128: the terminal source-edit classifier misread quoted `>` as a redirect and blocked every `tee`
- **Two false positives stranded a user mid-verification.** `shellWritesSource` in `lib/gate-guard-core.mjs` decides whether a `run_in_terminal` command is a source edit, and a `deny` here offers only one escape — "delegate to @fullstack" — which cannot fix a command that was never an edit. `git commit -m "renamed > foo.mjs"` split on raw whitespace into a bare `>` token followed by `foo.mjs"`; the quote-stripping in `isWriteTargetToken` turned that into `foo.mjs`, so an ordinary commit read as a redirect onto a source file and was denied. The same shape caught any command whose quoted argument happened to contain `> something.ext`.
- **`tee` was a source write regardless of its target.** `if (head === 'tee') return true` never inspected the filename, so `npm test 2>&1 | tee test-results.log` — a benign capture of test output during verification — was denied, while the `mv`/`cp` check two lines below already did the right thing by testing its actual argument.
- **The tokenizer is now quote-aware.** A new single-pass `tokenizeShellCommand` splits words on *unquoted* whitespace and emits the redirect operators (`>`/`>>`) and pipeline separators (`|`/`||`/`&&`/`;`) as their own tokens *only when unquoted* — so a `>` inside quotes is literal text, never a redirect. A lone `&` (as in `2>&1`) stays a word character and backslashes stay literal, so file-descriptor duplication and Windows paths survive intact. The redirect scan and the per-segment scan both run against this token stream.
- **`tee` now mirrors `mv`/`cp`:** flagged only when its actual argument names a write target. `npm test 2>&1 | tee lib/gate-guard-core.mjs` (a deliberately malicious target) is still a source write — the fix narrows false positives without weakening the fail-closed guarantee.
- **Proven against the broken code.** The new regression cases in `test/lib/gate-guard-core.default-deny.test.mjs` cover all four issue criteria plus the unquoted-redirect and session-artifact writes that must stay caught; the two false-positive cases were verified to fail on the pre-fix implementation and pass after, and every existing `shellWritesSource`/`isSourceEditTool` case still passes.

### Fixed - #105: the bug lane could not leave `lane-set` — the grill artifact's only writer could never fire
- **A user's bug lane wedged forever.** `@rubber-duck` completed a grill twice, `grill-result.json` never appeared, `approve plan` was refused ("does not accept event `start-impl`"), and the orchestrator — told nothing — concluded its agents were broken and began doing the work inline, the one thing its prompt forbids. On the bug lane `lane-set --finish-grill--> grill-done` is the ONLY pre-implementation transition and its sole evidence is that file.
- **The writer existed and was unreachable.** `projectWorkerReturn` wrote `grill-result.json` only when `validateGrillResult` passed, and that validator demands a FLAT body carrying `schemaVersion` and `returnedAt` — while `agents/rubber-duck.agent.md` instructed the agent to nest its body under `report` and never mentioned either field. An agent obeying its own card failed validation on ~11 fields. Three sources owned one shape and all three disagreed. `critique-result.json` carried the identical latent break on the feature lane.
- **The host now owns the machine fields.** `taskId`, `schemaVersion` and `returnedAt` are facts the host holds (task state, a constant, its own clock) and an agent cannot know; requiring them of a language model made every artifact a coin-flip. `lib/workflow/normalize-return.mjs` unwraps a `report` envelope and stamps all three, so the agent supplies only its analysis. `lib/workflow/agent-contracts.mjs` is now the single registry of shape → validator → artifact → example, and the tests enumerate IT rather than a hand-written list.
- **A chatty return no longer vanishes.** `extractEmbeddedJson` took the span from the first `{` to the last `}` in the agent's whole reply — fine for the terse `@router` return it was tested with, fatal for a grill report whose prose contains a brace ("the guard returns `{}` for anonymous callers"). The span parsed as nothing and the entire return was discarded: no worker-return file, no artifact, no gate move. It is now a string-aware balanced-brace scanner that finds every JSON candidate and prefers the one that names an agent.
- **Identity comes from the host, not from the model.** `SubagentStart` carries `agent_type`, and its `agent_id` is the `tool_use_id` of the dispatch that spawned the agent. `lib/hooks/subagent-index.mjs` records that join, so a return the model forgot to sign is still attributable — it used to be thrown away in silence. Self-reported `agentName` remains a fallback.
- **Silence is over.** Every failure path exited 0 with its complaint on a stderr channel the model never reads. A return that cannot become evidence now exits `EXIT_BLOCK` with a message naming the agent, the artifact NOT written, the exact missing fields, the gate that will therefore not advance, and the instruction to re-dispatch — *never* to work inline or hand-author the artifact, which would fabricate the very evidence the gate exists to check.
- **Artifacts are bound to their task.** `requireArtifact` and the dispatch gate's `readDiagnosisValid` checked existence and shape but never ownership, and nothing deletes `.devmate/state/*.json` between tasks — so a stale `diagnosis.json` went on authorizing a fix for a bug nobody had re-diagnosed. Both now compare `taskId`.
- **`grill_complete` / `critique_complete` are finally emitted.** Defined in the trace schema since E11-3, rendered by the trace viewer, promised on the agent card — and written by nothing. The tests proved the schema accepted the event and that `appendTraceEvent` could write it; neither proved anything ever called it. The grill vocabulary is unified on `assumptions` (the phantom `wrongAssumptions`, which no validator knew, is gone).
- **CI can now see this class of bug.** `scripts/check-artifact-graph.mjs` (new, in `npm run verify`) walks `gate --requires--> artifact --produced by--> contract --written by--> projector` and fails the build on a broken edge; `test/conformance/agent-contract-roundtrip.test.mjs` pushes the payload each agent's own card documents through the real projector and demands the artifact land and validate; `test/e2e/bug-lane-evidence.e2e.test.mjs` replays the whole bug lane through the real hooks, asserting the gate and the on-disk artifacts after EVERY turn. `check-contracts.mjs` could never have caught it: it validates `grill-result.json` only if it finds one, and nothing wrote it.

### Changed - #100: the session-artifact protection keys were enforced but unpublished — the shared contract now declares them (contract v4)
- **The guard's protection policy was invisible outside devmate.** #93 added two real config keys — `sessionArtifactPaths` (globs no agent may hand-edit) and `sessionArtifactWriters` (`{glob, agents[]}` exceptions) — validated by the hand validator and documented, but absent from docs/devmate-config.schema.json. The schema is `additionalProperties: true`, so nothing broke: a consumer got no completion and no red squiggle on a typo'd key, only a runtime config-load failure that denies every source edit; and monoroot, the sole writer of the merged multi-root config, had no basis to emit a key the shared contract never declared, so multi-root workspaces could not declare a session-artifact policy at all.

The descriptions state the protective default (.devmate/state/**, .devmate/session/**; spec.md → spec-writer) — an omitted key protects, it does not disable.
- **The producer can now write the policy, and fails closed on a malformed one.** monoroot validates both keys when present, propagates them from the primary repo like verification/testGlobs/maxConcurrentAgents, and drops a malformed policy rather than writing a config the consumer would refuse to load.
- **The shared fixtures corpus covers the new keys on both sides** — an accepted merged config carrying a policy, plus a rejected empty glob and a rejected writer entry with no agent roster (a half-read roster is a boundary widened by accident). All three are scope-tagged `both`, so the consumer's validator and the producer's predicate must agree on every one.
- **contractVersion is bumped to 4** (schema + corpus changed, so the pinned hash and both version constants move together, per the contract's own sync procedure); `schemaVersion` stays at 2, because both keys are additive.

### Fixed - #99: both persona boundaries were inert, and each one's comment pointed at the other as the layer that really enforced
- **Neither layer had ever run.** The edit-time rule (gate-guard Rule 5) needed a persona pinned on task state; nothing ever pinned one, so it was skipped on every call — and its comment rested the guarantee on the completion-time check. The completion-time check read the persona from the dispatch's tool_input, which reaches the hook elided to the literal string "..." — so its guard condition was undefined on every real dispatch, and the persona-scope check AND the TDD tripwire behind it were both skipped. Its docs, in turn, cited Rule 5. A frontend worker could edit a backend file, at every gate, with both layers green in CI.
- **The captured payload settles the design, and kills the edit-time rule outright.** A PreToolUse event carries no agent identity of any kind — session_id, tool_name, tool_input, tool_use_id, cwd, transcript_path, and nothing else. agent_type exists only on SubagentStart/SubagentStop. So an edit arriving at the guard cannot be attributed to one of several concurrent workers, by any mechanism: not a roster, not a scalar pin, and not a parent link — agent_id already IS one (it is the spawning runSubagent's tool_use_id), and it is useless here, because the edit event carries nothing to join it against. Rule 5 is deleted, not repaired. A rule that reads as a boundary while enforcing nothing is worse than no rule; a rule that guesses is worse still, because a non-persona string reaching ownsFile does not fail open — it denies every edit.
- **The per-worker boundary now lives, solely and honestly, at completion time — and actually fires.** The persona rides back on the worker's own returned contract (top-level persona, next to agentName), which is the only channel a captured payload proves reaches devmate.
