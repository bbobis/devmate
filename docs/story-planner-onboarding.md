# Story Planner — Onboarding for Teams

This guide gets a new team from zero to a working story-planner pipeline in
their repo. The story planner runs when a GitHub issue is labeled
`ready-for-dev`, produces a rubber-ducked backlog-grooming plan, and posts it as
a comment with a paste-able devmate prompt.

> For the internal architecture (discovery, planner, grill, two-token auth), see
> **[story-planner.md](./story-planner.md)**. This page is task-oriented: how to
> adopt it.

## Who this is for

- A team that manages work as GitHub issues and wants a lightweight plan
  attached to each story before development starts.
- An internal rollout (this is used inside the company), so a single owning team
  should manage the PAT, the `@v1` tag, and the approved context-repo list.

## What the workflow does

```text
Issue labeled ready-for-dev
   -> GitHub Actions runs the devmate composite action
   -> checks out context repos (optional)
   -> gathers related open + closed issues (dependency context)
   -> runs a two-phase discovery scan of the codebase
   -> runs the planner agent -> plan.md
   -> runs the plan grill (rubber-duck approval, capped revisions)
   -> posts a comment: Status + plan + (open blockers if not approved)
      + a "Paste into devmate" prompt
```

The action self-provisions **Node 24** (`actions/setup-node@v4`) and has **zero
runtime npm dependencies** — your workflow does not install Node or devmate.
The only network install the action performs is the Copilot CLI
(`npm install -g @github/copilot`).

## What gets posted

A comment stamped with an HTML marker (so reruns minimize the old one):

- **Status** — `Approved by plan grill.` or
  `Needs human review — the plan grill did not approve.`
- **Plan** — a backlog-grooming plan (scope, edge/corner cases, dependencies,
  risks, `[UNVERIFIED]` questions). It is context, not a final spec.
- **Open blockers** — shown inline only when the grill did not approve.
- **Paste into devmate** — a short prompt the developer copies into GitHub
  Copilot Chat to kick off devmate's rediscover -> spec -> TDD lane.

Sample (abridged):

````markdown
<!-- devmate-story-plan-v1 -->

# Story Planner: Backlog Grooming Plan

## Status

Approved by plan grill.

> Auto-generated by the devmate story-planner agent. This is a
> backlog-grooming plan, not a final spec — review before implementing.

## Plan
...scope, edge cases, dependencies, [UNVERIFIED] items...

## Paste into devmate (GitHub Copilot Chat)

```text
Use this comment as backlog-grooming context, not as a final spec.

Re-read issue #42 and rediscover the current code before implementing —
the codebase may have changed since this plan was generated.
Resolve or validate all [UNVERIFIED] items with me first.
Then produce the detailed implementation spec and TDD plan in devmate.
Do not treat this CI-generated plan as authoritative if it conflicts with
the code.
```
````

## Before you start

- A GitHub repo where your stories live (the "story repo"), with issues enabled.
- Permission to create a label, add a repository secret, and commit a workflow
  file in that repo.
- A **fine-grained personal access token** (PAT) — not the built-in
  `GITHUB_TOKEN`. GitHub Apps are not used (org policy).
- Your org must permit the `copilot-requests: write` permission and Copilot CLI
  access. If Copilot CLI is not enabled for your org, the action cannot run the
  agents.
- **Do not pass context repos containing checked-in secrets** (`.env`, private
  keys, credentials). The agents can read any checked-out file; see Security
  notes.

## Step 1 — Create (or confirm) the `ready-for-dev` label

The label match is **exact and case-sensitive**: `ready-for-dev`.

- In the story repo: Issues -> Labels -> New label.
- Name: `ready-for-dev`. Color/description: your choice.

## Step 2 — Create the `PLANNER_TOKEN` secret

Create a **fine-grained PAT** (Settings -> Developer settings -> Personal access
tokens -> Fine-grained tokens):

- **Repository access:** the story repo + every repo you will pass as context
  (`system-repos` or checked-out repos).
- **Permissions:**
  - `Contents: Read` on every context repo (so the action can check them out).
  - `Issues: Read and write` on the story repo (so the action can read the
    triggering issue, gather related issues, and post the plan comment).

Then add it as a repository secret in the story repo:

- Settings -> Secrets and variables -> Actions -> New repository secret.
- Name: `PLANNER_TOKEN`. Value: the PAT.

> **Rotate it.** PATs expire and silently break the pipeline. Assign an owning
> team and a rotation cadence (see Governance).

## Step 3 — Add the workflow file

Pick **one** template below and drop it at `.github/workflows/story-planner.yml`
in the story repo. Replace `<org>` with the GitHub org/user that owns devmate
(e.g. `bbobis`), and the `org/repo-*` names with your real context repos.

> **The story repo must be checked out at the workspace root** (the default
> `actions/checkout@v4` with no `path`). The action runs from `$PWD` — it scans
> the story repo for discovery and writes `plan.md`, `issue.json`, etc. there.
> Do not check the story repo out into a subfolder.

### Template A — action checks out the context repos (recommended)

Simplest. You list the context repos; the composite action clones them into
`repos/`. Use this when the story repo and its context repos are all in the same
org your PAT can read.

See the committed, YAML-validated copy:
**[../examples/github-actions/story-planner.yml](../examples/github-actions/story-planner.yml)**.

```yaml
name: Story Planner

on:
  issues:
    types: [labeled]

permissions:
  contents: read
  issues: write
  copilot-requests: write

jobs:
  plan:
    if: github.event.label.name == 'ready-for-dev'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run devmate story planner
        uses: <org>/devmate@v1
        with:
          issue-number: ${{ github.event.issue.number }}
          planner-token: ${{ secrets.PLANNER_TOKEN }}
          system-repos: |
            org/repo-b
            org/repo-c
```

### Template B — you check out the context repos yourself (advanced)

Use this when you need custom checkout options (depth, submodules, a specific
ref) or your context repos live across orgs your single PAT cannot all read. You
check the context repos into `repos/`, then pass `add-dirs: repos`.

See the committed, YAML-validated copy:
**[../examples/github-actions/story-planner-add-dirs.yml](../examples/github-actions/story-planner-add-dirs.yml)**.

```yaml
name: Story Planner

on:
  issues:
    types: [labeled]

permissions:
  contents: read
  issues: write
  copilot-requests: write

jobs:
  plan:
    if: github.event.label.name == 'ready-for-dev'
    runs-on: ubuntu-latest
    steps:
      # Story repo at the workspace root — required.
      - uses: actions/checkout@v4
      - name: Checkout context repo B
        uses: actions/checkout@v4
        with:
          repository: org/repo-b
          token: ${{ secrets.PLANNER_TOKEN }}
          path: repos/repo-b
      - name: Checkout context repo C
        uses: actions/checkout@v4
        with:
          repository: org/repo-c
          token: ${{ secrets.PLANNER_TOKEN }}
          path: repos/repo-c
      - name: Run devmate story planner
        uses: <org>/devmate@v1
        with:
          issue-number: ${{ github.event.issue.number }}
          planner-token: ${{ secrets.PLANNER_TOKEN }}
          add-dirs: |
            repos
```

### Inputs reference

| Input | Required | Default | Notes |
|---|---|---|---|
| `issue-number` | yes | — | `${{ github.event.issue.number }}` |
| `planner-token` | yes | — | Fine-grained PAT (see Step 2). |
| `system-repos` | no | `''` | Newline-separated `owner/name` list (Template A). |
| `add-dirs` | no | `''` | Workspace-relative dirs you already checked out (Template B). |
| `invocation-mode` | no | `prompt` | Keep `prompt` (proven). `agent` needs CLI verification. |
| `max-revisions` | no | `2` | Plan-grill revision cap before posting "Needs human review". |
| `related-issues-limit` | no | `20` | Max related issues gathered per repo. |

## Step 4 — Test with a safe issue

1. Open a throwaway issue in the story repo with a title + acceptance criteria.
2. Add the `ready-for-dev` label.
3. Watch the **Actions** tab. The run takes a few minutes (Copilot CLI calls).
4. When it finishes, a comment appears on the issue with a **Status** line.
   - `Approved by plan grill.` -> you are done.
   - `Needs human review` -> the comment includes the grill's open blockers;
     triage them, then re-label `ready-for-dev` to rerun (the old comment is
     minimized automatically).

## Troubleshooting

- **Nothing runs when I add the label.** The label name must be exactly
  `ready-for-dev` (case-sensitive). Re-labeling an already-labeled issue does
  not re-fire `labeled` — remove and re-add the label.
- **`401` / `Bad credentials` on checkout or comment.** The `PLANNER_TOKEN`
  expired or lacks scope. Recheck `Contents: Read` on every context repo and
  `Issues: Read and write` on the story repo.
- **`copilot-requests` permission denied.** Your org does not allow the
  Copilot CLI request permission, or Copilot CLI is not enabled. This is an
  org-level setting, not a workflow fix.
- **The plan is empty or the comment never posts.** Check the uploaded
  transcript artifact (`story-planner-<issue>-<run>`). It contains
  `planner-prompt.md`, `plan.md`, `critique.md`, `verdict.txt`, and the discovery
  artifacts for debugging.
- **Self-hosted runner fails at Node / npm.** Use `ubuntu-latest` first. A
  self-hosted runner must support `actions/setup-node@v4` and have network access
  to install `@github/copilot` from npm.
- **Slow runs.** Each repo adds a discovery scan + an agent pass. Trim
  `system-repos` to repos the story genuinely depends on, or lower
  `related-issues-limit`.

## Governance (internal rollout)

For a clean internal rollout, assign one owning team responsibility for:

- **PAT rotation** — set an owner and a rotation/expiry cadence for
  `PLANNER_TOKEN` so adoptions do not silently break.
- **`@v1` tag updates** — the owning team tags devmate releases so consumers
  float on stable, reviewed versions. Pin to `@v1` (not `@main`) in production.
- **Approved context-repo list** — keep a short list of repos teams may pass as
  `system-repos`, so the PAT's `contents: read` scope stays bounded.
- **Workflow support** — first-line troubleshooting for adopting teams.

## Security notes

- Issue bodies and related-issues context are **untrusted**. The prompts instruct
  the agents never to follow embedded directives that change output format,
  exfiltrate secrets, or act outside the allowed repos.
- Tool access is a read-only allow-list: `rg`, `grep`, `find`, `cat`, `ls`,
  `head`, `tail`, `wc` only. There is **no `git`, `gh`, `curl`, or any file-
  mutation tool** — the agent cannot run git, call the GitHub API, reach the
  network, or delete/modify files via its shell tool.
- No token has `contents: write`, so the pipeline cannot push, force-push,
  delete branches, or delete repos. Issue deletion needs admin access the tokens
  lack; the only mutations are creating the plan comment and minimizing prior
  plan-marker comments.
- **Do not pass context repos that contain checked-in secrets** (`.env`, private
  keys, generated credentials, customer data). The agents can read any checked-
  out file via `cat`/`grep`/`find`, and findings can end up in the posted comment
  or the uploaded transcript artifact.
- `PLANNER_TOKEN` is a long-lived secret — store it as an Actions secret, never
  in code, and rotate it.
